| KPI | 值 |
|---|---|
| **MFU** | **53.71 %** |
| **Step time** | **1742.6 s** |
| Compute time | 1560.0 s（89.5 %） |
| Comm time | 182.6 s（10.5 %） |
| Mem stall | 0 s |
| Idle | 0 s |
| Peak power（仅 GPU TDP 估算） | 256 kW |
| Confidence | 0.85 |
| Coverage | in_dist |
| Wallclock（单次仿真本身） | 52.5 s |
| Trace cache | `chakra/.../320d752dc7cbb10d/trace` |
| Provenance | `engine=astra-sim · version=2.1.0-analytical · selected_by=engine_preference` |

### 关键观察
- **计算占主导（89.5 %）**：1F1B 流水成功掩盖了大部分通信。EP=1 ⇒ 没有 all-to-all，只有 TP all-reduce + PP send/recv。
- **Comm/Compute = 11.7 %**：8 卡 NVLink-5 域内 TP all-reduce + 跨节点 PP send/recv 总开销约占一个 step 的 1/9，与 Llama 3.1 paper 公开的 405B 训练利用率级别吻合。
- **Mem stall = 0**：FP8 weight + selective recompute 把 activation 内存压住，没有 spill。
- **训练日数（按 15.6T tokens 估算）**：`15.6T / (4096 * 8192) tokens/step × 1742.6 s/step ≈ 9379 天`（≈ 25.7 年）。这是 256 卡的物理上限；Llama 3.1 paper 用的是 16k 卡。要把训练时间压到几个月需要扩到 4k–16k 卡或加大 batch。

## 4. 与 surrogate-analytical 对比（同一负载、不同候选策略）

`sim-fa7ec6` 流水线最终采纳的是 surrogate 跑出的 best 候选；将其与 astra-sim 跑用户配置的结果对比可以看出两个引擎的判读差异：

| 字段 | Astra-sim（用户策略 TP8 PP8 EP1 CP1） | Surrogate（pipeline best 候选） |
|---|---:|---:|
| MFU | 53.71 % | 54.56 % |
| Step time | 1742 600 ms | 119 127 ms |
| Compute | 1560 024 ms | 64 996 ms |
| Comm | 182 601 ms | 34 390 ms |
| Mem stall | 0 ms | 14 738 ms |

| Peak kW | 256 | 365 |
| Train days（按 15.6T tokens） | ≈ 9379 | 41.1 |

> Surrogate 的 `step_ms ≈ 119 s` vs Astra-sim 的 `step_ms ≈ 1743 s` 差 14.6 倍，主要是 surrogate 的 analytical 公式对 405B dense 这种巨型模型 step 时长做了 closed-form 估算，未抵到通信细粒度细节；astra-sim 走 chakra trace + cycle-accurate event sim，结果更可信。**校准后会用 astra-sim 修正 surrogate**（这正是 calibration-svc 的工作）。

## 5. 中途遇到的问题与解决

### Issue 1 — astra-sim 服务被自动 disable
**现象**：发起 run 前 `GET /v1/engines/astra-sim` 返回 `status: disabled`。

**根因**：`shared/engine_runtime/runtime.py:103-115` 心跳每 30 s 一次，`engine-registry-svc` 90 s 内没收到就 `UPDATE bs_engine SET status='disabled'`。心跳异常 message 为空（`heartbeat exception:`），怀疑是 astra-sim wrapper 进程跑 chakra 时阻塞 asyncio event loop 偶发心跳超时。

**临时解决**：`docker-compose restart astra-sim-svc` → boot 时自动重新 register，status 回到 active。验证 `last_seen_at` 实时更新即可。

**长期改进建议（未实施）**：
- 把心跳从同一 event loop 拆到独立线程，或在 chakra wrapper 旁边显式 `asyncio.shield`
- 把 `disable_stale` 阈值从 90 s 调到 180 s
- log heartbeat exception 的 `repr(exc)` 而不是 `str(exc)`，便于诊断

### Issue 2 — pipeline 不会把 user_override 锁死给 astra-sim
**现象**：通过 `POST /v1/runs` + `strategy_override` 发起的训练 run，`_engine_provenance.engine = "surrogate-analytical"`，不是预期的 astra-sim。

**根因**：`services/engine-svc/app/pipeline.py:138-163` 是「baseline + 5 候选 scan + top-k by MFU」结构。user_override 只是 5 候选之一，scan 里另外 4 个是 `SCAN_CANDIDATES`（含 EP=8/16、CP=2、ZBv2、Chimera 等），这些都不在 astra-sim envelope 内 → 都走 surrogate。最后按 MFU 排序，胜出的是 surrogate 的某条候选。

**当前规避**：直接调 `POST /v1/predict` 加 `engine_preference: "astra-sim"` 就走 astra-sim（已验证）。

**改进建议（待你确认是否实施）**：
- A. 给 `POST /v1/runs` 加可选 `engine_preference` 字段，run-svc 落到 `kpis._engine_preference`，engine-svc.predict 透传到 registry。配合 UI 「仿真引擎」选择器即可端到端选引擎。
- B. 训练流水线可选「仅跑用户策略，不做 scan」mode（适合用户已经定型策略只想看一次精确仿真）。

### Issue 3（未触发）— astra-sim 第三方代码
本次端到端跑通**未需要修改 astra-sim 第三方代码**。chakra wrapper 输出 `exit=0 elapsed=2745.9ms`，所有 256 个 sys（GPU rank）都正常 emit `[statistics] Wall time / Comm time / GPU time`，无 stderr。

## 6. 验证结论

✅ **C01 集群 + Llama-3.1-405B + Astra-sim** 端到端可用。
- envelope 严丝合缝（TP=8 PP=8 EP=1 CP=1 selective 1F1B FP8 B200 全部命中）
- 单次仿真 wallclock ~52 s，给出 cycle-accurate 的 step time / MFU / breakdown
- chakra trace cache 命中（hash `320d752dc7cbb10d`），后续相同负载会更快

⚠️ **目前 web UI 的「仿真引擎」选择器只是 envelope 校验，不会真正把请求 pin 到 astra-sim**。要真的让 UI 跑出 astra-sim provenance，需要按 Issue 2 改进建议改 run-svc / engine-svc / 前端的 engine_preference 透传链。

## 7. 附录 — 复现命令

```bash

  'engine_preference':'astra-sim',
  'payload': {
    'cluster':{'gpu_model':'B200','gpu_count':256,'electricity_usd_per_kwh':0.092,'pue':1.18},

  }

req = urllib.request.Request('http://engine-registry-svc:8089/v1/predict',
  data=body, headers={'content-type':'application/json'}, method='POST')
print(urllib.request.urlopen(req, timeout=120).read().decode())
"

# 3. 流水线 run（按 web UI 的逻辑发，含 scan）

  -H 'content-type: application/json' -d @path/to/run-payload.json
```
